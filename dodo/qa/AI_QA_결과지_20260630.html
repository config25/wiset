4. QA 발견 이슈

ID심각도제목내용 · 영향상태
ISSUE-1심각 type1 역량 JSON 구조 불일치 AI가 {그룹:{역량:{score,reason,sources}}} 3겹 객체로 응답하나, 파서는 {그룹:{역량:점수}} 2겹을 기대 → MismatchedInputException으로 역량 점수 전량 미적재. 리포트가 역량 분석 없이 생성됨. reason·sources 정보도 유실. 미해결
ISSUE-2중간 consulting_log 데이터 오염 한 사용자 계정에 타인 상담 9건(증권·제약·배터리 등 무관 주제 + 타인 이름·Zoom/개인 링크)이 적재됨. 쿼리는 user_sn 필터 정상 → 시드/테스트 데이터 오염으로 추정. AI 입력 품질 저하 및 운영 시 개인정보 외부 전송 리스크. 조사 필요
ISSUE-3경미 근무지·고용형태 중복 미제거 DB 중복 행이 조립 단계에서 dedup 없이 그대로 요청에 실림(예: 근무지 3개가 2회 반복). 코드 결함보다 데이터 중복. 현 동작은 회귀 테스트로 고정됨(D*/G). 데이터 정리

5. 권고 · 다음 단계

  • ISSUE-1 (우선): type1 응답 스키마 확정(3겹 객체가 서버 표준인지 확인) → 파서를 {그룹:{역량:{score,reason,sources}}} 구조로 맞춤. reason·sources 저장 여부에 따라 persist·TN 테이블 컬럼 검토 동반.
  • ISSUE-2: tn_cnsl_reqst_info의 user_sn별 건수 점검 → 오염 시드 정리 또는 클린 테스트 계정으로 재검증. 운영 전 개인정보 비식별 처리 점검.
  • ISSUE-3: 원천 데이터(sys_user_desired_region 등) 중복 행 정리. 필요 시 조립 단계 dedup 추가(추가 시 D*/G 회귀 테스트 갱신).
  • 로깅: 요청 프롬프트 원문 + unstructured_data 길이 로깅 적용됨 — 운영 환경에서는 개인정보 평문 로깅 정책(보존/접근) 재확인 권장.
